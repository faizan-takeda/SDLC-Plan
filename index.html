The Repo Decision · Beat 2 — Case Study

What a monorepo actually costs: ask Meta.

Meta started building its monorepo tooling roughly a decade ago — because publicly available source control could not, and still cannot, handle a repository of tens of millions of files. They didn't abandon the monorepo. They built everything it demanded:

Meta monorepo · 10M+ files · millions of commits
SaplingCustom source-control system (CLI + Mononoke server) — Git collapses at this scale.
EdenFSVirtual filesystem — checks out in seconds, fetches files only when accessed. A full checkout is infeasible.
Buck2From-scratch build system in Rust — traditional build tools can't handle large multi-language monorepos.
≈ 10 years of platform engineering · permanent dedicated teams · still evolving in 2025

Meta's design goal: operations scale with the files a developer touches — not with repo size. Achievable only with custom infrastructure and permanent staffing. A monorepo at scale is a multi-year platform-engineering commitment, not a repository setting. Takeda will not staff Sapling, EdenFS, and Buck2 — so an org-wide monorepo is off the table.

Phase 1 · The mandate — now

Justify every new initiative

No new initiative starts without answering one question: "Is this genuinely new, or an enhancement to a system that already exists?" If it's an enhancement, it improves the existing system — it does not spawn a parallel one. Duplication dies by rule, not by migration.

Phase 2 · The structure — earned

Per-system monorepo

Once the mandate holds, formalize it: one repo per system or program — never one for the org. Manageable with standard tooling, clean GxP boundaries per system, and the repo itself becomes the enforcement point: new work either enhances the system inside that repo, or must justify a new one.

Org-wide monorepo → Stop Per-system monorepo → Earn it Structure follows discipline — not the other way around.

Stopping is how we protect focus.

Internal Discipline

Every agent passes the same gate.
Genuine users — not demos.

Stopping the vendor is half the job. The other half is internal: no team builds an agent without passing four checks, and no agent scales without proving real users.

New DevX agent idea
Check 1

Reuse check

Can GitHub, Rovo, or Claude already do this? Prove the gap before proving the concept.

Check 2

Duplication scan

Does a similar agent already exist inside Takeda? Register it in the DevX registry — or don't build it.

Check 3

User reality

How many genuine users vs. red users? Repeated production usage — not launch-day logins.

Check 4

Refinement proof

Is this a platform — or just a prompt + an MCP connection + a workflow rule? If it can be config, it must be config.

Green — scale

Repeated active users · merged PRs · accepted artifacts · measurable time saved.

Amber — refine

Promising use case, but thin repeat usage or incomplete evidence. Iterate, re-measure.

Red — stop

Demo-only usage · duplicate idea · no owner · no measurable output · compliance risk.

The DevX Decision Board — Engineering, Architecture, Quality/GxP, Security, Product — runs this gate and classifies every idea Adopt / Configure / Build / Stop. That is the reuse mechanism this organization is missing.

The Pathway

Not a timeline. A gated decision system.

Work is never approved because a team presented first or is largest. Every agent, vendor scope, and internal build passes the same evidence gates. Progress by evidence, not by date.

G0 · Freeze

Pause duplicate spend

Pause new custom DevX builds, vendor expansion, and monorepo work. Capture active initiatives, teams, tools, spend, and overlapping use cases.
Exit → Inventory captured
G1 · Measure

Pull the free baseline

GitHub Copilot metrics, Rovo/Jira usage, PR cycle time, review time, adoption, active users. Native telemetry before funding anything new.
Exit → Real baseline exists
G2 · Classify

Registry: Adopt / Configure / Build / Stop

Every agent and initiative classified. Registry fields: owner, users, tooling, GxP status, duplication score, evidence of value.
Exit → Registry populated
G3 · Prove users

Separate adoption from excitement

Signal-rate every agent green / amber / red using real telemetry. Demo-only and duplicate work is stopped here.
Exit → Signal ratings assigned
G4 · Build

The Takeda layer — thin slice

One or two regulated workflow slices: traceability, audit pack, validation evidence, policy routing, human approval controls.
Exit → GxP thin slice proven
G5 · Decide

Scale, refine, stop — or narrow vendor role

No broad rollout without evidence: measured adoption, quality, cost, compliance burden, reusable patterns — and whether any vendor scope remains justified.
Exit → Decision paper · evergreen operating model live

Run by 15–20 people. Not 100. Not a vendor. Product owner · platform architect · GitHub/Copilot engineer · Atlassian/Jira engineer · Quality/GxP lead · security partner · 1–2 delivery engineers.

Adopt what the giants build. Configure what Takeda already owns. Build only the regulated layer. Stop what duplicates or can't prove real use.

Let the giants build the generic brain. Let Takeda build the regulated pharma brain. And before we spend like a transformation — let us measure like an engineering organization.
